# Delivery Guide スキル

マークダウンのデザインドキュメントを読み込み、実装・デリバリーまでの包括的なガイドを生成する。

---

## インプットの受け取り方

1. **セッション名から読み込む場合（推奨）**: ユーザーが `session-name` を指定した場合、`~/.claude-utils/session/<session-name>/design-doc.md` を読み込む。
2. **ファイルから読み込む場合**: アップロードされた `.md` ファイルを読み込む（`file-reading` スキルを参照）
3. **プロンプトから貼り付ける場合**: 会話中に貼り付けられたマークダウンテキストをそのまま使用

ドキュメントの粒度を判定してから生成を開始する:
- **機能仕様レベル (What)**: ユーザーストーリー・要件中心 → 技術的な判断を補完しながら生成
- **技術設計レベル (How)**: アーキテクチャ・API設計中心 → 設計に忠実に詳細化

---

## 生成する項目

以下の順で、ドキュメントの内容に基づいて生成する。該当しない項目は「該当なし（理由）」と明記してスキップ。

---

### 1. 前提条件・依存関係の整理

実装に着手する前に確認すべき事項を列挙する。

- 外部 API・サービスとの依存関係
- 他チーム・他機能との依存（先行して完了が必要なもの）
- 必要な環境変数・シークレット
- ライブラリ・パッケージの追加要否

---

### 2. 実装方法

ドキュメントの内容を実現するための具体的な実装手順。エンジニアがこのセクションだけを見て実装に着手できる粒度で記述する。

#### 2-1. アーキテクチャ概要

- アーキテクチャ上の判断と根拠（複数案がある場合は比較表を提示）
- 主要なコンポーネント・モジュールと責務
- コンポーネント間のデータフロー（入力 → 処理 → 出力の流れ）
- 技術スタック・使用するライブラリの選定理由

機能仕様レベルのドキュメントの場合は、技術的な判断を補完して提示し、採用した理由を明記すること。

#### 2-2. ファイル単位の変更内容

変更・追加するファイルを一覧にし、それぞれの役割と変更概要を示す。

フォーマット:
```
| ファイルパス | 操作 | 変更概要 |
|---|---|---|
| src/api/routes/example.ts | 新規 | エンドポイント定義 |
| src/services/example.ts | 新規 | ビジネスロジック |
| src/models/example.ts | 変更 | フィールド追加 |
```

#### 2-3. API・インターフェース設計

新規・変更が必要なAPIエンドポイントやインターフェースを定義する。

- エンドポイント一覧（メソッド・パス・概要）
- リクエスト / レスポンスの型定義（コードブロックで明示）
- 内部モジュール間のインターフェース（関数シグネチャ・イベント定義など）
- 既存APIへの破壊的変更がある場合はその影響範囲

#### 2-4. 主要ロジックの実装方針

複雑なビジネスロジックや判断が必要な処理について、擬似コードまたは実コード例で示す。

- 主要な処理フローのステップバイステップ解説
- 条件分岐・状態遷移がある場合はその整理
- 外部サービス連携の呼び出し方法・データ変換ロジック
- 実装上のトレードオフや注意点

#### 2-5. 実装順序

依存関係を考慮した推奨実装順序をステップで示す。各ステップには以下を含む：

- 対象ファイル・コンポーネント
- 完了条件（何ができたらこのステップは完了か）
- 前のステップへの依存関係

#### 2-6. データモデル・スキーマの変更

- 新規テーブル・コレクション・フィールドの定義
- 既存スキーマへの変更点（追加・削除・型変更）
- マイグレーション要否と手順の概要
- インデックスの考慮

#### 2-7. エラーハンドリング方針

デザインドキュメントに記載がない場合でも、実装上必要な観点を補完する。

- 想定されるエラーケースと対処方法
- ユーザーへのエラー表示方針（メッセージの粒度・多言語対応）
- リトライ・フォールバック戦略
- エラーのログ出力レベル（DEBUG / INFO / WARN / ERROR）

---

### 3. テスト・動作確認

#### 単体テスト
- テスト対象のコンポーネント・関数
- モックが必要な外部依存
- 境界値・異常系のテストケース例

#### 結合テスト
- テスト対象のインターフェース（API・イベント・DB）
- テストデータのセットアップ方針

#### E2E テスト（該当する場合）
- ユーザーフロー単位のシナリオ
- テスト対象環境（ステージング推奨）

#### ローカル環境での動作確認
- セットアップ手順（環境変数・モックサービスの準備）
- 確認シナリオと期待結果
- よくあるハマりどころ

#### ステージング環境での動作確認
- デプロイ後に確認すべきシナリオ
- ローカルとの差異（本物の外部サービス連携など）

---

### 4. 非機能要件

実装に関連する非機能要件を以下の観点で整理する。

#### セキュリティ・権限
- 認証・認可の変更点（新規エンドポイント・権限ロールの追加など）
- 入力バリデーションが必要な箇所
- 機密データの取り扱い（マスキング・暗号化）
- OWASP 観点での懸念点（XSS・CSRF・SQLインジェクションなど）

#### パフォーマンス
- N+1 クエリが発生しうる箇所
- 大量データ処理が想定される処理とその対策
- キャッシュ戦略の要否
- 負荷テストが必要かどうかの判断

#### 監視・ログ・アラート
- 新規で追加すべきログ出力箇所
- 監視ダッシュボードへの追加項目
- アラートの閾値・通知先の変更要否

---

### 5. デプロイ・ロールバック

#### デプロイ手順
- デプロイ順序（DB マイグレーション → バックエンド → フロントエンド など）
- フィーチャーフラグ・段階リリースの要否
- ダウンタイムが発生しうる操作
- デプロイ後に確認すべきメトリクス・ログ

#### ロールバック手順
- ロールバックのトリガー条件（何が起きたら戻すか）
- ロールバック手順（コード・DB マイグレーションそれぞれ）
- データ整合性への影響とリカバリ方法

---

### 6. PO と事前確認すべき項目

デザインドキュメントに曖昧さや仕様の抜け漏れがある箇所を列挙し、実装着手前に合意を取るべき事項をまとめる。

フォーマット:
```
## PO確認事項

| # | 確認内容 | 背景・理由 | 優先度 |
|---|---------|-----------|--------|
| 1 | ... | ... | 高 / 中 / 低 |
```

優先度の基準:
- **高**: 合意がないと実装方針が決まらない
- **中**: 実装は進められるが、後から変更コストが高い
- **低**: 細部の確認。実装後でも対応可能

---

## 出力フォーマット

- マークダウン形式で出力
- 各セクションに見出し（`##`）を付ける
- 「該当なし」の項目は理由を1行で添えてスキップ
- コードブロックは言語を明示する（例: ` ```sql `, ` ```typescript `）
- PO確認事項は必ずテーブル形式で出力する

### ファイル保存

セッション名から読み込んだ場合は、生成した Delivery Guide を同じセッションディレクトリに保存する：

```
~/.claude-utils/session/<session-name>/delivery-guide.md
```

コンソールにも全文を出力する。

---

## 注意事項

- デザインドキュメントに記載のない技術的判断を補完する場合は、その旨を明示する（例: 「※ドキュメントに記載なし。一般的な慣習として以下を推奨:」）
- 実装者が1人であることを前提とした粒度で記述する
- フロントエンド・バックエンドを問わず対応するが、ドキュメントのスコープ外の実装は推測で書かない
