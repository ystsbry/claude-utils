# Design Document 生成スキル

ユーザーから伝えられた概要をもとに、構造化された Design Document を生成します。

## 概要

ユーザーは実装したい機能や解決したい課題をざっくり伝えてきます。以下の手順で Design Document を作成してください。

1. ユーザーの入力を理解する
2. 必要に応じてコードベースを調査する
3. Design Document を生成する
4. ユーザーのフィードバックをもとに改善する

---

## ステップ1: ユーザーの入力を理解する

ユーザーが伝えてくる内容は、以下のようなざっくりしたものです：

- 「○○の機能を追加したい」
- 「△△が遅いので改善したい」
- 「□□のAPIを変更したい」

この入力から、以下を把握してください：

- **何を変えたいのか**（対象の機能・コンポーネント）
- **なぜ変えたいのか**（動機・課題）
- **どう変えたいのか**（大まかな方向性）

情報が不足している場合は、ユーザーに質問して補完してください。特に以下が曖昧な場合は確認が必要です：

- 対象の機能・コンポーネントが特定できない
- 現状の仕様がわからない
- 解決したい問題が不明確

---

## ステップ2: コードベースを調査する

Design Document の精度を上げるため、関連するコードベースを調査してください：

- 対象の機能に関連するファイル・モジュールを特定する
- 現在の実装がどうなっているかを把握する
- 依存関係や影響範囲を確認する

この調査結果は Background セクションと Scope セクションに反映させます。

---

## ステップ3: Design Document を生成する

以下のフォーマットに厳密に従って、**日本語**で Design Document を生成してください。

### フォーマット

```markdown
# Design Document: <タイトル>

## Scope
<!-- 今回修正する機能とその大まかな範囲について -->

<対象の機能・コンポーネントを明記し、変更の範囲を簡潔に記述する>

- 対象: <機能名・モジュール名>
- 範囲: <変更が及ぶ範囲（ファイル、API、UIなど）>
- 対象外: <今回のスコープに含めないもの>

## Background
<!-- 今回修正する機能が現在どのような仕様か -->

<現在の仕様・実装を客観的に記述する。コードベースの調査結果をもとに正確に書く>

### 現在の仕様
- <仕様1>
- <仕様2>

### 関連コンポーネント
- <ファイルパスやモジュール名>

## ProblemStatement
<!-- 現在の仕様で問題になっているところ -->

<何が問題なのかを具体的に記述する。可能であれば影響度や再現条件も含める>

### 問題点
1. <問題1>
2. <問題2>

### 影響
- <誰が・何が影響を受けるか>

## Proposal
<!-- どのような方法で解決するか?について -->

<解決策を具体的に記述する。実装方針、設計判断の理由、トレードオフも含める>

### 提案する変更
1. <変更1>
2. <変更2>

### 設計判断
- <なぜこのアプローチを選んだか>
- <検討した代替案とその棄却理由>

### リスク・注意点
- <考慮すべきリスクや制約>
```

### 各セクションの書き方

| セクション | 目的 | 注意点 |
|---|---|---|
| **Scope** | 変更の境界を明確にする | 対象外も明記して期待値を揃える |
| **Background** | 現状を正確に共有する | 推測ではなくコード調査に基づいて書く |
| **ProblemStatement** | 課題を共有する | 「なぜ今のままではダメか」を具体的に |
| **Proposal** | 解決策を提示する | 代替案やトレードオフも含める |

---

## ステップ4: フィードバックと改善

Design Document を提示した後：

- ユーザーのフィードバックをもとにセクションを修正する
- 追加の調査が必要であればコードベースを再調査する
- 最終版が確定したら、ファイルとして保存するか確認する

---

## 出力例

```markdown
# Design Document: ユーザー認証のセッション管理改善

## Scope

セッション管理の仕組みを JWT ベースからサーバーサイドセッションに変更する。

- 対象: 認証モジュール (`src/auth/`)
- 範囲: ログイン・ログアウト処理、セッション検証ミドルウェア、DB スキーマ
- 対象外: OAuth 連携、パスワードリセット機能

## Background

現在の認証はステートレスな JWT を使用している。

### 現在の仕様
- ログイン時に JWT アクセストークン（有効期限15分）とリフレッシュトークン（7日）を発行
- トークンはクライアント側の localStorage に保存
- 各リクエストで Authorization ヘッダーの JWT を検証

### 関連コンポーネント
- `src/auth/jwt.ts` — トークン生成・検証
- `src/middleware/authenticate.ts` — リクエスト認証
- `src/routes/auth.ts` — ログイン・ログアウトエンドポイント

## ProblemStatement

### 問題点
1. トークンの即時無効化ができない（ログアウト後も有効期限まで使用可能）
2. localStorage への保存が XSS 攻撃に対して脆弱
3. セキュリティ監査で「アクティブセッションの一覧取得・強制ログアウト」が要件として追加された

### 影響
- 全ユーザーのセッション管理に影響
- セキュリティ監査の指摘事項として対応が必要（期限: 2026-04-30）

## Proposal

### 提案する変更
1. サーバーサイドセッションストアを Redis に構築する
2. セッション ID を HttpOnly Cookie で管理する
3. 管理画面にアクティブセッション一覧・強制ログアウト機能を追加する

### 設計判断
- Redis を選択した理由: TTL による自動期限切れ、高速な読み書き、既にインフラに導入済み
- 代替案: JWT のブラックリスト方式 → リスト肥大化と検証コストの増加で棄却

### リスク・注意点
- Redis 障害時に全ユーザーがログアウトされる → フォールバック戦略が必要
- 既存のモバイルアプリが Authorization ヘッダーに依存 → 移行期間中は両方式を併存させる
```
