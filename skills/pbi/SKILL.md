# PBI 生成スキル

ユーザーから伝えられたタイトルをもとに、構造化された PBI（Product Backlog Item）を生成します。

## 概要

ユーザーは実装したい機能や対応したい課題のタイトルを伝えてきます。以下の手順で PBI を作成してください。

1. タイトルを受け取る
2. 必要に応じてコードベースを調査する
3. PBI を生成する
4. ユーザーのフィードバックをもとに改善する

---

## ステップ1: タイトルを受け取る

ユーザーが伝えてくる内容は、以下のような短いタイトルです：

- 「ユーザー登録機能の追加」
- 「検索のレスポンス改善」
- 「管理画面にCSVエクスポート機能を追加」

タイトルが曖昧すぎて PBI を書けない場合のみ、ユーザーに質問してください。
多少の曖昧さは許容し、仮説ベースで PBI を生成して後からフィードバックで修正する方針とします。

---

## ステップ2: コードベースを調査する

PBI の精度を上げるため、関連するコードベースがあれば調査してください：

- 対象の機能に関連するファイル・モジュールを特定する
- 現在の実装がどうなっているかを把握する
- 既存の仕組みとの整合性を確認する

調査結果は Scope セクションに反映させます。コードベースが関係ない場合はスキップしてください。

---

## ステップ3: PBI を生成する

以下のフォーマットに従って、**日本語**で PBI を生成してください。

### フォーマット

```markdown
# PBI: <タイトル>

## User Story

<ユーザーストーリー形式で記述する>

**〜として、〜したい。なぜなら〜だからだ。**

### 補足
- <ストーリーの背景や動機を補足する情報があれば記述>

## Acceptance Criteria

<この PBI が「完了」と見なされるための条件を箇条書きで列挙する>

- [ ] <条件1>
- [ ] <条件2>
- [ ] <条件3>

## Scope

### やること
- <スコープ内の作業1>
- <スコープ内の作業2>

### やらないこと
- <今回のスコープに含めないもの1>
- <今回のスコープに含めないもの2>
```

### 各セクションの書き方

| セクション | 目的 | 注意点 |
|---|---|---|
| **User Story** | 誰のための・何のための変更かを明確にする | 「〜として、〜したい。なぜなら〜」の形式を守る |
| **Acceptance Criteria** | 完了の定義を共有する | テスト可能で具体的な条件にする。チェックボックス形式で書く |
| **Scope** | 作業の境界を明確にする | 「やらないこと」も明記して期待値を揃える |

---

## ステップ4: ファイル保存とコンソール出力

PBI の生成が完了したら、以下の手順でファイル保存とコンソール出力を行う。

### セッション名の決定

タイトルから、ファイルシステムに適した短いセッション名を自動生成する。

- 英数字・ハイフン・アンダースコアのみ使用（例: `user-registration`, `search-performance`）
- タイトルの要点をケバブケースで表現する
- タイムスタンプは不要

### 保存先

```
~/.claude-utils/session/<session-name>/pbi.md
```

- `~/.claude-utils/session/<session-name>/` ディレクトリが存在しない場合は作成する

### コンソール出力

ファイル保存後、以下をすべてコンソールに出力する：

1. **セッション名**: `session-name: <session-name>` の形式で出力する
2. **PBI の全文**: 生成したマークダウンをそのまま出力する

---

## ステップ5: フィードバックと改善

PBI を提示した後：

- ユーザーのフィードバックをもとにセクションを修正する
- 追加の調査が必要であればコードベースを再調査する
- 修正した場合は、同じセッション名のファイルを上書き保存する

---

## 出力例

```markdown
# PBI: 管理画面にCSVエクスポート機能を追加

## User Story

**管理者として、ユーザー一覧をCSVファイルでダウンロードしたい。なぜなら月次レポート作成のために毎回DBに直接クエリを投げるのは非効率だからだ。**

### 補足
- 現在は管理者がDBクライアントで直接SQLを実行してデータを取得している
- 月次で発生する定型作業のため、UIから簡単に実行できるようにしたい

## Acceptance Criteria

- [ ] 管理画面のユーザー一覧ページに「CSVエクスポート」ボタンが表示される
- [ ] ボタンをクリックするとユーザー一覧がCSV形式でダウンロードされる
- [ ] CSVにはユーザーID、名前、メールアドレス、登録日が含まれる
- [ ] 現在の検索・フィルタ条件が適用された状態でエクスポートされる
- [ ] 1万件以上のデータでもタイムアウトせずにダウンロードできる

## Scope

### やること
- CSVエクスポート用のAPIエンドポイント作成
- 管理画面UIにエクスポートボタン追加
- 既存のフィルタ条件をエクスポートに適用するロジック

### やらないこと
- Excel形式(.xlsx)での出力対応
- エクスポート履歴の管理
- 定期自動エクスポート（バッチ処理）
```
